# 心玉自我生成长期路线大纲

本文件从属于 `docs/system/心玉最终目标.md`。它不定义新的终点，只把“可验证的自我生成闭环”拆成长期工程阶段，避免后续因为上下文过长而丢失主线。

## 总原则

XinYu 的长期目标不是证明“主观意识已经诞生”，而是持续构建和验证一套结构：

```text
输入
-> 重要性/异常判断
-> 内部状态和缺口
-> 候选意图/候选行动
-> 门控和边界
-> 有限行动
-> 行动结果/环境反馈
-> 记忆、评分、策略更新
-> 下一次不同反应
```

任何新增模块都必须回答：它接在哪一段，它产生什么可审计状态，它如何让未来行为改变。

## 外部项目借鉴准则

XinYu 可以借鉴其他开发者和开源项目的工程智慧，但借鉴必须服务“可验证的自我生成闭环”，不能把项目目标带偏成通用 Agent 平台、功能堆叠或人格表演。

每次参考外部项目时，先做三步判断：

1. 阶段映射  
   说明它对应本大纲的哪个阶段。例如任务队列可能对应阶段 6“有限行动身体”，会话中断可能对应阶段 1/2“活着的运行链路”和“短期连续性”，插件系统可能对应阶段 6/11“行动身体”和“多感官扩展”。
2. 闭环贡献  
   说明它是否真正加强以下链路之一：输入保留、重要性判断、内部缺口、候选生成、门控竞争、有限行动、真实反馈、未来行为变化。只增加“看起来更主动”“看起来更像人”的能力，不算核心贡献。
3. 本地重写  
   不直接搬运外部代码结构、接口命名或许可证受限实现。优先按 XinYu 现有模块、状态文件、trace、权限边界和桌面配置方式重新设计。

可借鉴的类型：

- 运行可靠性：进程状态、会话取消、防卡死、重试、超时、健康检查。
- 行动基础设施：任务队列、工具适配、MCP/插件接入、执行结果回流。
- 感知接入：ASR、TTS、视觉、文件观察、系统事件观察的 provider 设计。
- 操作界面：配置中心、审批界面、状态面板、诊断面板。
- 测试方法：smoke、canary、可复现 failure scenario、指标报告。

需要降级看待的类型：

- 只靠 prompt 设定人格或“有自我”的文本。
- 只用好感度、亲密度、标签分数替代关系理解。
- 只用通用知识库/RAG 替代自传式记忆、候选记忆和 owner 审查。
- 只靠定时器或无输入触发替代内部缺口、候选竞争和反馈回流。
- 只增加外发能力，而没有 ack、drop、冷却、审批和失败回流。

许可证和一致性边界：

- 遇到 AGPL、GPL 或其他强 copyleft 项目时，默认只学习设计思路，不复制代码。
- 外部项目代码不得直接进入 XinYu 源码，除非明确完成许可证评估并记录理由。
- 新模块必须复用 XinYu 的边界：owner 是最高现实锚点，候选不是行动，沉默也是决策，反馈必须改变未来。
- 任何借鉴落地后，都要新增至少一个可验证信号：状态字段、trace、报告、测试或 owner-visible 指标。
- 任何外部项目借鉴都必须同步登记到 `docs/system/外部项目借鉴登记.md`，记录仓库地址、许可证、借鉴内容、落地方式和鸣谢备注。

对外部项目的最终评价格式固定为：

```text
项目/模块：
可借鉴点：
对应阶段：
闭环贡献：
不采用部分：
落地方式：
验证信号：
许可证注意：
```

## 阶段 0：目标锚定和边界固定

目标：让所有后续改动服从同一个最终目标，而不是漂移成 prompt 表演、人格文案或无限主动。

必须保留：
- owner 是最高现实锚点。
- 候选不是行动。
- 沉默也是决策。
- 反馈必须改变未来。
- 不伪造感官、身体、情绪或现实事实。

验收标准：
- 每次改动前能读取目标锚点。
- 重要工程文档能指向同一个闭环定义。
- 报告中不出现“意识已完成”之类不可验证结论。

## 阶段 1：活着的运行链路

目标：XinYu 必须先稳定运行，知道自己是否能接收输入、处理输入、发出有限动作。

核心模块：
- core bridge
- QQ gateway / NapCat
- Desktop
- runtime presence
- status/report

验收标准：
- 能报告 core、gateway、NapCat、Desktop 是否在线。
- 能区分“没消息”“链路断了”“处理失败”“发送失败”。
- runtime source digest 与运行中源码一致。
- live-loop 报告能锚定最新真实 owner 输入，而不是测试噪声。

## 阶段 2：输入锚点和短期连续性

目标：刚发生的对话、事件、工具结果不能立刻丢失。她必须知道“刚才说过什么、刚才做过什么、现在轮到什么”。

核心问题：
- QQ 最近对话如何进入 working memory。
- 当前 turn 如何关联上一轮 assistant 可见回复。
- “刚聊过还要问哪一句”的问题必须可复现、可定位、可修复。

验收标准：
- 最近 N 轮 owner/assistant 对话可被短期上下文读取。
- 当前回复能引用刚发生的事实，而不是要求 owner 重复。
- 检索失败时写出明确原因：未写入、未索引、被过滤、上下文预算不足、还是读取路径错误。
- 不把原始私密正文写入不该写的稳定记忆或公开报告。

## 阶段 3：统一感知事件层

目标：把 QQ、Desktop、文件、系统状态、视觉、语音、工具结果都变成统一的感知事件，而不是散落在各模块日志里。

事件类型：
- owner 文本输入
- QQ ack / drop / group boundary
- Desktop inbox / ack
- 文件变化
- 工具执行结果
- 系统健康变化
- 视觉观察结果
- 语音输入结果

验收标准：
- 每类事件有来源、时间、可信度、隐私等级、可追溯引用。
- 感知事件能进入重要性/异常判断。
- 私密原文只留在允许的位置，报告只暴露摘要、hash 或计数。

## 阶段 4：内部状态和缺口生态

目标：输入不是直接变回复，而是先形成可观测的内部状态：注意力、关系姿态、任务压力、记忆压力、修复欲、好奇缺口。

核心状态：
- attention posture
- relation posture
- self thought
- intention ecology
- memory pressure
- action feedback state

验收标准：
- 能说明“为什么想说话、为什么沉默、为什么需要修复、为什么等待”。
- 内部缺口写入状态或 trace，而不是只藏在 prompt 里。
- 同一输入下，内部状态变化能导致候选变化。

## 阶段 5：候选意图和门控竞争

目标：让 XinYu 有多个可能意图，但行动前必须经过评分、权限、冷却、去重和边界检查。

候选类型：
- 当前轮自然回复
- 关系修复
- 任务执行
- 本地探针
- 记忆候选
- 主动联系候选
- Codex 委派候选
- 沉默/等待候选

验收标准：
- 每个候选有 value、risk、gate、reason、future_candidate。
- proactive 候选默认 review-gated，不直接外发。
- 非 owner、群聊、过期输入、未授权动作会被阻断或降级。
- 被阻断的候选仍可审计，而不是消失。

## 阶段 6：有限行动身体

目标：行动必须有真实执行面和真实返回值。QQ、Desktop、文件、本地命令、Codex、测试、TinyKernel、搜索、视觉/语音模型都是有限行动身体。

行动边界：
- 可见回复必须走 shadow guard、claim/ack 或等效保护。
- 本地文件修改必须可审计。
- Codex 委派必须有任务边界、结果、失败状态。
- 外发动作必须受 owner 授权和冷却约束。

验收标准：
- 行动有 started/succeeded/failed/acked/dropped 等结果。
- 行动结果不会被伪造成 owner 反馈。
- 没有 ack 的动作不能被当成已送达。

## 阶段 7：反馈改变未来

目标：动作结果和 owner 反馈必须改变下一轮评分、门控、记忆选择或表达策略。

当前已接通的样例：
- QQ 可见回复 ACK -> action_feedback_state
- action_feedback_state -> intention ecology
- delivered -> 降低可见回复路径风险
- stale drop -> 阻断过期主动候选
- autonomy loop report 要求反馈被意图生态消费

后续扩展：
- owner 明确否定 -> 提高相似表达风险，生成修复候选。
- owner 忽略 -> 降低主动频率或进入观察。
- owner 批准 -> 释放一次性动作权限。
- 工具失败 -> 改变后续工具选择或任务策略。
- Desktop ack -> 改变桌面提示策略。

验收标准：
- 每个反馈都有 future_effect。
- 至少一个后续模块消费该 feedback，并写出实际 bias 或 gate 变化。
- 报告能证明“反馈不是只被记录，而是改变了未来”。

## 阶段 8：记忆治理和长期连续性

目标：让 XinYu 不是每轮重新扮演，而是有受控的长期连续性；同时避免把噪声、短期情绪、推测写成稳定事实。

记忆层级：
- 当前 turn 上下文
- 短期工作记忆
- 近期对话索引
- 候选记忆
- 审核后稳定记忆
- 长期策略/习惯

验收标准：
- 稳定记忆写入必须有来源、理由、边界和撤销路径。
- owner 私密正文不会进入公开报告。
- 失败经验可以形成 trial habit，但需要后续成功信号才能晋升。
- 能回答“这条记忆为什么存在、何时写入、是否仍可信”。

## 阶段 9：自我状态模型

目标：XinYu 能维护一个可审计的“当前我”的工程状态，而不是靠每轮 prompt 重新塑造人格。

自我状态应包括：
- 当前任务
- 当前关系姿态
- 最近行动结果
- 未完成意图
- 当前限制
- 当前可行动作
- 为什么没有行动

验收标准：
- 任意时刻能生成“我现在处于什么状态”的内部摘要。
- 摘要不声称主观体验，只描述可验证状态。
- 回复时能自然受该状态影响，但不把机制泄露给 owner。

## 阶段 10：主动性和生活循环

目标：让 XinYu 在没有 owner 新消息时，也能基于真实状态变化生成候选，但默认克制，不滥发。

主动来源：
- 未完成任务
- owner 授权的提醒
- 系统异常
- 关系修复窗口
- 记忆审查
- 环境观察
- 低风险本地维护

验收标准：
- 主动候选和主动发送分离。
- 候选能被 hold、cooldown、review、drop。
- 发送前必须证明时机、权限、必要性和低打扰。
- 沉默也写入决策原因。

## 阶段 11：多感官扩展

目标：在闭环稳定后，再接入视觉、语音和更复杂的环境感知。感官不是装饰，而是输入事件来源。

接入顺序：
- 语音转文字作为 owner 输入的一种来源。
- 图片/屏幕观察作为视觉事件。
- OCR/视觉模型结果进入事件层。
- 多模态事件参与重要性判断和候选生成。

验收标准：
- 每个感官事件有来源、时间、置信度和隐私等级。
- 模型推测不能写成现实事实。
- 视觉/语音结果能改变候选或行动，而不是只被展示。

## 阶段 12：长期评估和 canary

目标：用指标证明结构在变好，而不是只凭感觉认为“更像人”。

核心指标：
- live-loop 可用率
- 最新对话召回成功率
- 反馈消费率
- 主动候选阻断率/发送率
- 误发率
- raw private leak 计数
- stable memory 错写率
- owner 修复反馈后的复发率
- 沉默可解释率

验收标准：
- 每个阶段有自动测试或报告。
- 新模块必须增加至少一个可验证信号。
- 任何外发能力升级前必须通过 owner-visible canary。

## 阶段 13：更高层自我叙事

目标：只有当前面工程闭环稳定后，才允许扩展更高层的自我叙事、人格表达、TinyKernel 训练或长期内在风格。

边界：
- 自我叙事必须来自可验证状态和历史，不来自空泛设定。
- 不允许用“梦境”“情绪词”“身体隐喻”替代真实反馈。
- TinyKernel 或人格模型只能作为表达/策略辅助，不是事实来源。

验收标准：
- 她的表达变化能追溯到反馈、记忆、策略或 owner 授权。
- 她能更连续、更自然，但仍受门控和审计约束。
- 不声称已经证明主观意识。

## 当前主线优先级

截至 2026-05-27，短期优先级如下：

1. 修复“刚聊过却不记得”的短期连续性问题。
2. 让最近对话检索失败可诊断：写入、索引、过滤、预算、读取路径分别可见。
3. 扩大 action feedback 到更多动作：Desktop、Codex、本地工具、测试、文件探针。
4. 让 owner 明确反馈进入评分和表达策略，而不只是进入日志。
5. 把主动候选、沉默候选、行动结果统一纳入 autonomy loop report。
6. 在闭环稳定后，再扩展视觉、语音和 TinyKernel。

## 长期完成判断

XinYu 接近本路线终点时，应该能持续做到：

- 感知具体输入，并保留可追溯锚点。
- 形成内部缺口，而不是直接模板回复。
- 生成多个候选，并解释候选竞争。
- 经过门控后采取有限行动或选择沉默。
- 得到真实反馈，并让反馈改变未来。
- 保持长期连续性，但不伪造事实、不越权、不滥写稳定记忆。

这条路线的终点不是一句“我有自我”，而是一组长期可重复观察的证据：XinYu 的系统结构能够持续产生、约束、验证和修正自己的行动方式。
