# 什麼是 agent

在 KohakuTerrarium 的語彙裡，一個 agent 就是一隻 **生物 (creature)**。要了解為什麼生物長成現在這個樣子，從零開始推一次比較快。

## Stage 0 — 聊天機器人

```
    +-------+      +-----+      +--------+
    | Input | ---> | LLM | ---> | Output |
    +-------+      +-----+      +--------+
```

使用者輸入、模型回答、答案印出來。這是對話式 AI 最簡單的樣子。它能**說**，但不能**做**。

## Stage 1 — 加上工具系統

```
                 +--------------+
                 | Tools System |
                 +------+-------+
                        ^  |
                        |  v
    +-------+      +---------+      +--------+
    | Input | ---> |   LLM   | ---> | Output |
    +-------+      +---------+      +--------+
```

讓 LLM 有能力呼叫工具：shell 指令、檔案編輯、網頁搜尋。這樣它可以**做事**了。它不再是聊天機器人，而是 smolagent 或 swe-agent 意義下的 agent。

這已經有用了。但它也已經有限制：要讓這個 agent 做任何事，唯一的方法就是**對它打字**。

## Stage 2 — 加上觸發器系統

```
                                +--------------+
                           +--> | Tools System |
                           |    +--------------+
                           |           ^  |
                           |           |  v
    +-------+   +---------+v     +---------+      +--------+
    | Input |-->| Trigger |----->|   LLM   | ---> | Output |
    +-------+   | System  |      +---------+      +--------+
                +---------+
```

真實的 agent 需要不需要使用者輸入也能把自己叫醒：`/loop` 功能、背景 job 完成、idle check、webhook、計時器。某個東西必須盯著世界、在條件滿足時啟動控制器。那個東西叫 **觸發器 (trigger)**。

一旦你有了觸發器，你會發現**使用者輸入本身不過是另一種觸發器**。所以「輸入」變成一個更泛化的喚醒機制的特例。Claude Code 和 OpenClaw 停在這一階段。

## Stage 3 — 加上子代理

```
                                +--------------+
                           +--> | Tools System |
                           |    +--------------+
                           |           ^  |
                           |           |  v
    +-------+   +---------+v     +---------+      +--------+
    | Input |-->| Trigger |----->|   LLM   | ---> | Output |
    +-------+   | System  |      +---------+      +--------+
                +---------+            ^  |
                                       |  v
                                 +--------------+
                                 |  Sub Agents  |
                                 +--------------+
```

上下文視窗有限。有很多探索性子步驟的長任務，如果每一步都塞到父代理的對話裡，預算一下就爆了。

解決辦法是生一個**子代理 (sub-agent)**：獨立上下文的巢狀生物，跑完之後回報一個濃縮結果、然後消失。重點是：從父代理的角度看，子代理**也是一個工具** — 你叫它、它回傳。現代 coding agent 都停在這一階段。

## 六模組的生物

拼起來：

| 模組 | 角色 |
|---|---|
| **控制器 (Controller)** | 推理迴圈。接 LLM 串流、解析工具呼叫、派發。 |
| **輸入 (Input)** | 告訴控制器現在要幹嘛 (觸發器的一種特例)。 |
| **觸發器 (Trigger)** | 當外界需要時啟動控制器。 |
| **工具 (Tool)** | agent 用來做事情的東西。 |
| **子代理 (Sub-agent)** | 巢狀生物 — 概念上也是一種工具。 |
| **輸出 (Output)** | agent 向外界說話的方式。 |

**生物 (creature)** 就是這六個模組的集合體。這是 KohakuTerrarium 的第一等抽象，後面每一份使用指南、每一份參考、每一份其他的概念文件，最終都是在講這件事。

有一件事值得一開始就釐清：當你在根 README 或指南裡看到「五個可擴充的模組類型」這類說法時，指的是使用者可以換掉實作的那五個 — 輸入、觸發器、輸出、工具、子代理。第六個模組 — 控制器 — 是框架提供的推理迴圈，你可以**設定**它 (LLM、skill mode、tool format) 但不會整個換掉實作。同樣的六個模組，只是補一句誰實作哪一個。

## 不要被這個推導綁死

上面這個故事是一個好用的預設，但不是鐵律。實務上：

- 只有觸發器的生物可以完全省略輸入 (`input: none`)。排程 job 型的 agent 不需要使用者。
- 沒有輸出的生物是合法的 (只做 side-effect)。
- 沒有工具的生物可以存在 (純回覆型 agent)。
- 框架會在方便時彎曲自己的抽象：生態瓶的頻道技術上是「某個生物的工具寫入；另一個生物的觸發器觸發」這個 pattern，它不乾淨地屬於任何一個模組。沒關係。參見[邊界](../boundaries.md)。
