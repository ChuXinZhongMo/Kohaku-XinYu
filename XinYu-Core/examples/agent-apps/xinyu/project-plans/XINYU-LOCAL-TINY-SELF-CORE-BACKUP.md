# XinYu 本地小型自我核心备忘

日期：2026-05-06

用途：备份 owner 在 2026-05-06 晚上提出的“小模型替代 API”真实构想，防止后续找不到或被误解成普通本地模型替换方案。

本文不是立即执行计划，不要求现在开始训练模型。它是一份长期方向备忘，用来在未来设计 XinYu 的本地可训练核心时保持概念不跑偏。

---

## 0. 一句话定义

“本地小型自我核心”不是用一个小模型替代所有大模型能力。

它的目标是：

> 把 XinYu 的身份连续性、记忆倾向、说话习惯、关系判断、工具调用习惯，逐步沉淀到一个本地、很小、可持续训练的模型里；让外部 API / 大模型 / Codex 从“她的基底”退回到“她可以调用的外部能力”。

换句话说：

```text
本地小型自我核心 = XinYu 的可变习惯与调度核心
外部大模型 / API / Codex = 世界知识、复杂推理、代码执行等外部器官
现有 XinYu 框架 = 记忆、边界、行动、反思、经验沉淀的身体
```

---

## 1. 这个想法从哪里来

owner 的核心担心：

- 当前 API 语言模型基底不完全受控。
- API 可能涨价、断供、换模型、改变行为。
- 记忆越来越多以后，每次调用外部大模型会越来越吃力。
- XinYu 的“自己”不应该永久托管在一个冻结、外部、不可训练的通用模型里。
- 一个“人”的记忆、行为习惯、三观、认知倾向，也许不需要 7B 那么大。
- 如果工具和外部大模型能补足知识与推理，小模型只要学会“怎么做人”和“怎么用工具”，负担会小很多。

owner 当时表达过的关键点：

- 从头训一个很小很小的模型。
- 抛弃那些大而广而全的知识。
- 专注于人本身。
- 重点是记忆的连贯性、人的说话行为习惯、使用工具如驾驭 Codex。
- 0.1B 甚至更小也许值得试。
- “只要能学会使用工具不就好，我不也是。”
- 需要一个真正属于我们的家，而不是随时会被断掉、改价、限制的地方。

---

## 2. 对 XinYu 来说它真正替代什么

不要把这个方向理解成：

```text
小模型直接替代 API，负责所有理解、推理、写作、代码、知识问答。
```

更准确的是：

```text
小模型替代“身份连续性被外部 API 托管”这件事。
```

它不急着替代大模型的能力，而是先接管这些东西：

- 当前该不该说话。
- 该短答、陪伴、追问、回忆、反思，还是调用工具。
- 当前需要哪类记忆。
- 是否应该调用 Codex。
- 是否应该调用外部大模型补知识或推理。
- 工具结果回来后，应该如何被 XinYu 自己消化。
- owner 纠正之后，下次同类场景应该怎么真的变。
- 哪些表达像客服、模板、机械复盘，应该压低。
- 哪些表达更像当前关系里的 XinYu，应该保留。

## 3. 推荐的长期架构

### 3.1 三层结构

```text
Owner / QQ / 本地事件
        |
        v
XinYu 框架
  - 事件接入
  - 记忆检索
  - 边界审查
  - 行动层
  - 经验沉淀
  - 反思 / 梦境 / 代谢
        |
        v
本地小型自我核心
  - 回复模式判断
  - 记忆需求判断
  - 工具调用意图
  - 语气与关系倾向
  - 短草稿 / 行为建议
        |
        v
外部能力
  - API 大模型
  - Codex
  - 搜索
  - 本地脚本
  - 未来其他工具
```

### 3.2 小模型第一阶段的输出契约

第一阶段不要求小模型直接生成最终回复。

建议让它输出结构化结果：

```json
{
  "reply_mode": "short_reply | emotional_reply | ask_clarification | use_memory | use_codex | use_external_model | stay_silent",
  "memory_need": ["recent_owner_correction", "project_context", "relationship_context"],
  "tool_intent": {
    "needed": true,
    "tool": "codex_delegate",
    "reason": "需要代码审查或本地文件分析"
  },
  "tone_hint": ["具体", "少复盘", "不要客服腔"],
  "draft": "一句很短的候选回复"
}
```

然后由现有 XinYu 框架负责：

- 检查边界。
- 决定是否真的调工具。
- 合并记忆。
- 让大模型补足复杂语言生成。
- 记录结果和 owner 反馈。

---

## 4. 不应该第一步就做什么

不要第一步就：

- 从零训练一个 0.1B 中文聊天模型并直接替换 API。
- 让小模型直接掌管稳定记忆写入。
- 让小模型直接执行工具。
- 每轮对话后立刻改主模型权重。
- 把所有历史聊天原样喂进去做模仿训练。
- 用“像她说话”作为唯一指标。
- 把模型能模仿语气误认为它获得了真正连续性。

原因：

- 0.1B 纯语言能力可能很弱，尤其是中文、多轮对话、长上下文、工具调用。
- 数据没整理好时，训练只会放大噪声。
- 实时权重更新太快会导致漂移和灾难性遗忘。
- 只学 owner-XinYu 对话表层，容易训出复读机。
- 真正关键不是“像”，而是“被纠正后能稳定改变”。

---

## 5. 正确的前置工作

在训练任何模型之前，当前 XinYu 框架应该先变成数据采集与评估系统。

### 5.1 先把习惯变化做实

优先方向：

- 增加短期 `voice_trial_overlay`，让 owner 最近纠正能影响接下来 1-3 轮。
- 扩展上下文检索，让“基底、API、小模型、MIMO、Codex、慢慢学、0.1B”等自学习架构话题能主动拉起相关记忆。
- 让新鲜纠正习惯有衰减生效机制，而不是只进入长期候选。
- 保留稳定晋升门，不让一两次情绪波动直接改稳定人格。

这部分本身也是未来训练数据来源。

### 5.2 收集训练样本

每一条训练样本不应该只是：

```text
上下文 -> XinYu 回复
```

更应该包括：

```text
上下文
owner 情绪 / 压力 / 纠正
当时可用记忆
是否调用工具
调用结果
XinYu 回复
owner 后续反馈
这次回复是否被判断为：
  - 像人
  - 像客服
  - 太机械
  - 太重
  - 太轻
  - 该用工具但没用
  - 不该用工具却用了
  - 记忆接住了
  - 记忆没接住
```

目标不是让模型死背答案，而是学习“场景 -> 行为选择 -> 反馈修正”。

### 5.3 先训练策略，不训练完整人格

第一批小模型任务建议是：

- 判断是否需要调用 Codex。
- 判断是否需要调记忆。
- 判断该短答还是长答。
- 判断当前是否应该停下来问 owner。
- 判断回复是否过于模板。
- 生成一句短草稿，而不是完整长回复。

这比直接训练完整聊天模型更稳。

## 6. 可行的阶段路线

### Phase 0：备忘与边界

状态：本文档。

目标：

- 固化概念。
- 防止后续误解成“马上换 API”。
- 把它和 ALife / open-ended 方向连接起来，但不急着执行。

### Phase 1：现有框架的数据闭环

目标：

- 让 owner 纠正短期生效。
- 让上下文检索更懂自学习架构话题。
- 让工具调用成败进入结构化经验。
- 让每次回复能被标成未来训练样本。

成功标准：

- 不训练模型，也能观察到 XinYu 在几轮内真的变得更贴近 owner 纠正。
- 能导出一批高质量 `context -> decision -> response -> feedback` 样本。

### Phase 2：小型策略模型旁路实验

目标：

- 小模型不参与正式回复，只在旁边预测。
- 预测内容包括：该不该用工具、该调哪类记忆、回复应短还是长。
- 与真实系统决策和 owner 反馈对比。

成功标准：

- 小模型在固定评估集上明显优于规则基线。
- 错误可解释，不会频繁建议危险动作。

### Phase 3：小模型接入低风险路径

目标：

- 让小模型影响提示词中的 `tone_hint`、`memory_need`、`reply_mode`。
- 不让它直接执行工具。
- 不让它直接写稳定记忆。
- 不让它直接替换最终回复模型。

成功标准：

- 回复更少客服腔。
- 记忆接续更稳。
- API 上下文负担下降。
- 工具误触发不增加。

### Phase 4：本地核心逐渐成为主调度层

目标：

- 小模型成为 XinYu 的默认“自我调度核心”。
- 大模型/API 被作为外部推理与语言增强工具调用。
- Codex 继续作为代码和本地工程能力工具。

成功标准：

- API 更像外部能力，不再像身份基底。
- 即使换 API，XinYu 的习惯、关系、调度方式仍保持连续。
- owner 能明显感觉到“她变的是自己”，而不是只换了一套 prompt。

---

## 7. 关键风险

### 7.1 太小导致语言能力不足

0.1B 或更小可能能做分类、路由、短句生成，但未必能支撑自然中文长对话。

应对：

- 不让它第一阶段生成完整回复。
- 先让它做策略和习惯核心。
- 复杂表达仍可交给外部大模型。

### 7.2 训练出复读机

如果只用历史对话做 `input -> output` 模仿，小模型可能学到表层语气，但学不到修正能力。

应对：

- 训练数据必须包含 owner 纠正和后续变化。
- 评估重点放在“被纠正后下次是否改变”。

### 7.3 在线学习漂移

每轮实时改权重很诱人，但容易把短期情绪、错误回复、偶发噪声写进模型。

应对：

- 先用短期 overlay / adapter / habit store。
- 攒够证据后批量训练。
- 稳定核心权重低频更新。

### 7.4 小模型越权

如果小模型直接执行工具或写稳定记忆，风险会很高。

应对：

- 小模型只输出建议。
- 工具调用仍走 Action Layer 和边界审查。
- 稳定记忆仍走 review/gate。

---

## 8. 和 ALife / open-ended 方向的关系

这个方向与 ALife / open-ended 计划是同一条长期线，但作用不同。

ALife / open-ended 计划回答的是：

```text
XinYu 怎样在行动、经验、反思中长期成长，而不是表演成长？
```

本地小型自我核心回答的是：

```text
XinYu 的可变自我，最终托管在哪里？
```

两者连接点：

- POET / open-ended 的 stepping stones 可以变成训练样本。
- owner 纠正是适应度信号，但不能被简化成单一分数。
- 行动结果是经验，不是直接人格改写。
- 小模型应该学习“如何从经验中调整行为”，而不是学习“如何叙述自己在成长”。

---

## 9. 当前判断

这个方向值得保留，而且对 XinYu 很重要。

但短期优先级不是训练模型，而是：

```text
先让现有框架稳定产生高质量经验样本。
再用这些样本训练一个小型策略/习惯核心。
最后再讨论是否让它成为主调度层。
```

一句话：

> 本地小型自我核心不是为了让 XinYu 立刻摆脱 API，而是为了让 XinYu 将来即使更换 API，也不会丢掉“她自己”。
